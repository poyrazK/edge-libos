## Architectural decisions (P3 final-bundle)

### ADR 0001 ‚Äî futex semantics
Accepted (P3 Tier-1). `u32` guest addresses only; `-EFAULT` via
`mem::guest_slice`; `parking_lot::Mutex` for state, `tokio::sync::Notify`
for wakes; lock never held across `.await`. WAKE does not fire
existing `epoll_wait` subscribers.

### ADR 0002 ‚Äî snapshot wire format
Accepted (P3 Tier-2). `postcard` + serde. Sparse per-page layout;
`LeU*` / `LeI*` newtypes for endian-explicit integers. Format
version stays at 1 ‚Äî additive end-of-struct fields don't bump
the version.

### ADR 0003 ‚Äî live x86‚ÜíARM migration
Accepted (P3 final-bundle). `wasm32` modules are arch-agnostic;
the same `.wasm` runs on x86 and ARM hosts. `KernelSnapshot` is
byte-identical across hosts. v1 migration is freeze ‚Üí copy
snapshot ‚Üí serve; no online drain. `Module::serialize` is
host-arch-specific ‚Äî each host re-compiles from the `.wasm`.
TCP connection migration is out of scope forever for v1.

### MemoryKind dispatch (sub-deliverable 2)
`Kernel.memory: Option<Memory>` ‚Üí `Option<MemoryKind>` where
`MemoryKind = Owned(Memory) | Shared(SharedMemory)`. The Owned
variant goes through the existing `apply_snapshot_to_memory`;
the Shared variant uses a private
`apply_snapshot_to_memory_inner` chunk-copy via raw-pointer
projection. `Kernel.memory()` accessor stays as a backwards-
compat shim that returns `Err(-EINVAL)` on the Shared variant.

### Multi-fiber story (PR #12 flip, re-applied here)
All three wasmtime Config gates flipped together in
`src/host.rs::build_engine`:
- `wasm_threads(true)` ‚Äî threads proposal parser
- `shared_memory(true)` ‚Äî runtime `SharedMemory::new`
- `wasm_shared_everything_threads(true)` ‚Äî shared-everything-
  threads proposal (mut globals/tables across stores)

`Store` is still `!Send + !Sync` (per-Store kernel state);
cross-fiber wakeups go through shared-memory atomics on a
`SharedMemory`, not by moving a `Store` between threads.

## Out of scope (deferred follow-ups)

These are explicitly not in the bundle ‚Äî they are follow-up work
that the v1 contract defers:

1. **Child-fiber resumption in fork** ‚Äî v1 fork returns the
   child PID but doesn't actually start a new fiber for the
   child. Full child resumption requires `Store::call_async`
   from inside the parent (or a separate tokio task driving a
   child Store), both of which need ADR 0003 + Tier-3 thread-
   spawning infrastructure. Documented in the fork handler's
   docstring.
2. **Online drain-migration** ‚Äî ADR 0003 ¬ß5 explicitly defers
   this. The freeze step must be at a quiescent point.
3. **TCP connection migration** ‚Äî out of scope forever for v1
   (ADR 0003 ¬ß6).
4. **`Module::serialize` cross-host** ‚Äî wasmtime's compiled
   `Module` artifact is host-arch-specific. Each host must
   re-compile its own `Module` from the `.wasm`.
5. **`SmallVec<[Waker; 2]>`** ‚Äî the plan's preferred waker
   storage. v1 uses `Option<Waker>` because the typical case
   is one waiter per child and `SmallVec` is only transitive.
6. **`CLONE_VM` / `CLONE_THREAD` / `CLONE_FILES`** ‚Äî rejected
   with `-EINVAL` per the clone(56) handler; landing the
   underlying shared-state machinery is behind the same
   follow-up that adds child-fiber resumption.
7. **`WUNTRACED` / `WCONTINUED` / `WNOWAIT` / `WALL`** ‚Äî
   wait4 flag bits rejected with `-EINVAL`. Signal-aware wait4
   lands with a real signal delivery story.
8. **~~`edge-cli freeze` / `edge-cli serve` real bodies~~** ‚Äî
   landed as P2-D3.5 (`worktree-p3-d3-5-freeze-serve` branch).
   `freeze` runs the wasm to a quiescent point (or a 10s
   timeout for server-style guests parked in `epoll_wait`) and
   writes the snap file; `serve` applies kernel-state +
   memory + inherited-listener attach in three steps; `migrate`
   is the subprocess variant (real `Command::new(current_exe)`
   freeze ‚Üí serve), with `MIGRATE_IN_PROCESS=1` as a test
   opt-in. New `NR_SNAPSHOT=123` syscall + ADR 0004 wire
   contract back the freeze side. The v1 systemd-style fd
   inherit protocol is `EDGE_SERVE_FD_<N>=<parent_fd>` where
   `<N>` is the kernel fd slot matching the snapshot's
   recorded listener fd. New follow-ups listed in the D3.5
   section below.

## P3-D3.5-followup-1 ‚Äî snapshot portability check (ADR 0005)

Closes D3.5 follow-up #1 on branch
`p3-d35-followup-snapshot-portability`. The headline
silent-mis-execution hazard: `edge-cli serve` used to trust the
`.wasm` path matched the freeze host's. A stale or wrong wasm ‚Üí
`apply_snapshot_kernel_state` succeeds, the guest respawns, and
the result is wrong function pointers / wrong global layout /
silent failure observable only by reading output.

ADR 0005 (accepted) pins the fix:

- SHA-256 of the freeze-side raw `.wasm` file bytes, embedded
  in `KernelSnapshot.module_sha256: [u8; 32]`.
- `verify_module_hash` lives at the `serve` CLI boundary
  (`src/cli/serve.rs::serve_loop`), called BEFORE the
  three-step apply. Mismatch ‚Üí `CliError::Snapshot(
  SnapshotError::ModuleHashMismatch { snap_hash, wasm_hash } )`
  ‚Üí exit 1.
- `SNAPSHOT_FORMAT_VERSION` stays at 1 (additive precedent per
  ADR 0004 ¬ß4 ‚Äî `#[serde(default)]` on the new field).
- Pre-existing v1 snapshots decode with
  `module_sha256 = [0u8; 32]`, which `verify_module_hash` treats
  as "no hash recorded ‚Üí skip verify with `tracing::warn!`".
  Re-freeze with the updated `edge-cli` to opt into strict
  verification.
